# HintBook Eval Pipeline (Python)

Standalone offline pipeline for iteratively refining HintBook prompts. Runs a matrix of `(prompt variant × model × state × dataset example)`, scores against ground-truth labels aggregated across every configured US-state driver-license dataset, and surfaces failure clusters to drive the next prompt revision.

By default every step sweeps every state in `runner.py::TARGETS` and aggregates the results into a single report — so prompt proposals are tuned for the cross-state average, not any one jurisdiction. The improved SYSTEM prompts are document-type-agnostic; per-state expert text is auto-loaded from `src/hints/thinking/<page_id>.txt` and injected into the user message at runtime.

Independent of the app — does NOT go through `server.js` or `vite.config.js`. Reads hint pages directly from `../src/hints/*.js` and `../src/hints/thinking/*.js` (parsed with `json5`, no Node runtime needed) and calls DeepInfra over HTTPS.

## Setup

```bash
python -m venv eval/.venv
source eval/.venv/bin/activate
pip install -r eval/requirements.txt

# .env already has DEEPINFRA_API_KEY (used by the app); reused here.
```

## Quickstart

```bash
python eval/run_pipeline.py                                     # full loop: baseline → score → propose → A/B → compare
# or run the steps by hand:
python eval/runner.py --variant v0_baseline                     # all states × all models
python eval/metrics.py                                          # latest run summary (aggregated + per-state)
python eval/iterate.py --base v0_baseline --proposals 3         # generates document-agnostic SYSTEM revisions
python eval/runner.py --variant v1_<auto_slug>                  # A/B a generated proposal across all states
python eval/metrics.py --all                                    # side-by-side comparison
```

Pass `--page <id>` to `runner.py` to restrict a sweep to one state (debugging only — proposals should still be generated against the multi-state baseline).

## Analysis notebook

`eval/analysis.ipynb` is the read side of the pipeline — it never calls the API or writes files, it just visualizes runs:

1. Headline metrics + per-model breakdown
2. Per-check confusion matrix (heatmap)
3. Per-section accuracy (bar chart)
4. **Failure spot-check** — inline image with the model's bbox overlay, finding sentence, and ground-truth label
5. Run-over-run trend (line chart across variants)
6. Variant diff on a shared example

Open it with your editor's Jupyter integration (VS Code, Cursor, JupyterLab) and pick the `eval/.venv` Python interpreter as the kernel — `ipykernel` is installed for you. The first cell is the imports; the second picks a run (latest by default; edit `RUN = ...` to pin one). Run top-to-bottom.

The notebook is generated by `eval/_make_notebook.py` — to change cells, edit that script and re-run it (`python eval/_make_notebook.py`). Don't hand-edit the `.ipynb` — JSON diffs poorly in git and your changes will be overwritten the next time someone regenerates.

## Dataset conventions

- **Partial labels**: only label checks you can verify with certainty. Unlabeled checks are silently skipped in scoring, so you don't have to hand-rate all 50 per image.
- **`meta.category`**: `"genuine"` or `"forged"` — drives verdict-accuracy metric.
- **`meta.tampered`**: list of check IDs that were intentionally altered. Used for tamper recall/precision.
- See `datasets/ca_dl/example_001/README.md` for a worked example.

## Metrics

| Metric | What it answers |
|---|---|
| `per_check_accuracy` | Of labeled checks, how often did the model match the label? |
| `tamper_recall` | Of `meta.tampered` checks, how often did the model flag NO/WARN? |
| `tamper_precision` | Of checks the model flagged NO, how often were they actually tampered? |
| `verdict_accuracy` | Does derived verdict bucket match `meta.category`? |
| `unverifiable_rate` | How often does the model dodge with UNVERIFIABLE? |
| `latency_ms_p50` / `tokens_total_sum` | Throughput / cost axis |

## Iteration loop

1. `python eval/runner.py --variant v0_baseline` — sweeps every state in `runner.py::TARGETS`
2. `python eval/metrics.py` — confirm baseline numbers (overall + per-state)
3. `python eval/iterate.py --base v0_baseline --proposals 3`
   - Clusters lowest-accuracy checks by `(page_id, section_id, expected → predicted)` so a single state can't drown out cross-state patterns
   - Sends aggregated + per-state metrics, state-tagged failure clusters, and the current SYSTEM prompt to the meta-prompter
   - Meta-prompter is hard-constrained to produce DOCUMENT-AGNOSTIC SYSTEM prompts (no state names, document types, or check IDs) — per-state expert text stays auto-loaded from `src/hints/thinking/<page_id>.txt`
   - Writes `prompts/v{n}_<auto-slug>.py` × N
4. `python eval/runner.py --variant v1_*` (re-runs across all states)
5. Compare `runs/<old>/report.json` vs `runs/<new>/report.json` — promote winner

Keep a held-out slice (~20% of each state's dataset) untouched until you're picking between finalists, so meta-prompting doesn't overfit to the visible set.
